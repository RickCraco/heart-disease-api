# Heart Disease Prediction API

A machine learning API that predicts the likelihood of heart disease from patient clinical data. The project demonstrates an end-to-end workflow: data ingestion, model training with hyperparameter tuning, model persistence, and serving predictions through a REST API with input validation, API key authentication, rate limiting, and automated tests.

> **Note:** This is an educational portfolio project. It is not intended for clinical or diagnostic use.

## Relation to Final Year Project (University of Hertfordshire)

This repository builds on the machine learning work carried out for my **Final Year Project (FYP)** at the [University of Hertfordshire](https://www.herts.ac.uk/). The FYP focused on exploring and evaluating models for heart-disease risk prediction from clinical features. This project takes that foundation further by packaging a tuned XGBoost pipeline behind a production-style FastAPI service — including authentication, rate limiting, health checks, automated tests, and Docker deployment — so the research prototype can be exercised as a real inference API.

## Overview

The application trains an XGBoost classifier on a heart disease dataset and exposes prediction and health endpoints via FastAPI. Preprocessing (scaling and one-hot encoding) is bundled inside a scikit-learn pipeline, so the same transformations are applied consistently during training and inference. The API is protected with an API key and rate-limited, and can be run locally or via Docker.

## Features

- **End-to-end ML pipeline** — download data, preprocess features, train, evaluate, and save the model
- **Hyperparameter tuning** — `GridSearchCV` with 5-fold cross-validation
- **REST API** — `POST /predict` with Pydantic validation and `GET /health` for readiness checks
- **API key authentication** — requests to `/predict` require a valid `X-API-Key` header
- **Rate limiting** — 10 requests per minute per client IP on `/predict` (via slowapi)
- **Structured logging** — separate log files for training and API inference
- **Docker support** — containerised deployment with built-in health check
- **Automated tests** — API validation, authentication, health, and utility tests with pytest

## Getting Started

### Prerequisites

- Python 3.10 or higher
- `pip`
- (Optional) Docker, for containerised runs

### Installation

Clone the repository and create a virtual environment from the project root:

```bash
git clone <repository-url>
cd heart-disease-api
python -m venv venv
```

Activate the virtual environment:

```bash
# Windows
venv\Scripts\activate

# macOS / Linux
source venv/bin/activate
```

Install dependencies:

```bash
# Runtime only
pip install -r requirements.txt

# Runtime + tests
pip install -r requirements-dev.txt
```

### Configuration

Create a `.env` file in the project root with your API key:

```env
API_KEY=your-secret-api-key
```

The key is loaded via `api/config.py` (pydantic-settings) and required for authenticated prediction requests.

### Train the Model

Training downloads the dataset if it is not already present, runs grid search, evaluates on a hold-out test set, and saves the model to `models/xgboost_pipeline.joblib`.

Run from the project root:

```bash
python -m src.train
```

Training metrics and configuration are written to `logs/training.log`.

### Run the API

Start the server from the project root (the trained model must exist first, and `.env` must define `API_KEY`):

```bash
uvicorn api.main:app --reload
```

The API will be available at `http://127.0.0.1:8000`. Interactive documentation is available at:

- Swagger UI: `http://127.0.0.1:8000/docs`
- ReDoc: `http://127.0.0.1:8000/redoc`

### Run with Docker

Build and run the container (ensure the model artifact and `.env` are available in the build/run context as needed):

```bash
docker build -t heart-disease-api .
docker run --env-file .env -p 8000:8000 heart-disease-api
```

The image exposes port `8000` and defines a Docker `HEALTHCHECK` against `GET /health`.

### Run Tests

```bash
pip install -r requirements-dev.txt
pytest
```

For verbose output:

```bash
pytest -v
```

Tests override authentication and disable rate limiting where appropriate so endpoint behaviour can be asserted in isolation.

## API Reference

### Authentication

`POST /predict` requires the header:

```
X-API-Key: <your-api-key>
```

Missing or invalid keys return HTTP `401`.

### Rate limiting

`POST /predict` is limited to **10 requests per minute** per client IP. Exceeding the limit returns HTTP `429`.

### `GET /health`

Reports whether the trained model was loaded successfully at startup.

**Success response** (`200`):

```json
{
  "status": "ok"
}
```

If the model failed to load, the endpoint returns HTTP `503`.

### `POST /predict`

Predict whether a patient is likely to have heart disease.

**Request body**

| Field | Type | Description |
|-------|------|-------------|
| `Age` | int | Age in years (0–120) |
| `Sex` | string | `"M"` or `"F"` |
| `ChestPainType` | string | `"TA"`, `"ATA"`, `"NAP"`, or `"ASY"` |
| `RestingBP` | int | Resting blood pressure in mm Hg (0–200) |
| `Cholesterol` | int | Serum cholesterol in mg/dl (0–600) |
| `FastingBS` | int | Fasting blood sugar > 120 mg/dl: `1` (true) or `0` (false) |
| `RestingECG` | string | `"Normal"`, `"ST"`, or `"LVH"` |
| `MaxHR` | int | Maximum heart rate achieved (60–202) |
| `ExerciseAngina` | string | Exercise-induced angina: `"Y"` or `"N"` |
| `Oldpeak` | float | ST depression induced by exercise (-3.0–7.0) |
| `ST_Slope` | string | `"Up"`, `"Flat"`, or `"Down"` |

**Example request**

```bash
curl -X POST "http://127.0.0.1:8000/predict" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-secret-api-key" \
  -d '{
    "Age": 58,
    "Sex": "M",
    "ChestPainType": "ASY",
    "RestingBP": 140,
    "Cholesterol": 289,
    "FastingBS": 1,
    "RestingECG": "ST",
    "MaxHR": 110,
    "ExerciseAngina": "Y",
    "Oldpeak": 2.0,
    "ST_Slope": "Flat"
  }'
```

**Example response**

```json
{
  "prediction": 1,
  "predicted_proba": 0.87
}
```

| Field | Type | Description |
|-------|------|-------------|
| `prediction` | int | Predicted class: `0` (no heart disease) or `1` (heart disease) |
| `predicted_proba` | float | Estimated probability of heart disease (0.0–1.0) |

Invalid input returns HTTP `422` with validation details. Prediction failures return HTTP `500`.

## Model Pipeline

1. **Preprocessing** — numerical features are standardised; categorical features are one-hot encoded (`drop="first"`)
2. **Classifier** — XGBoost, tuned via grid search over `n_estimators`, `max_depth`, `learning_rate`, and `gamma`
3. **Evaluation** — 70/30 stratified train/test split; metrics include accuracy (cross-validation), classification report, and ROC AUC on the test set
4. **Persistence** — the full fitted `GridSearchCV` object is serialised with joblib

## Limitations

- Trained on a single public dataset; performance may not generalise to other populations
- API key auth is a basic shared-secret check; no user accounts or token rotation
- Model and logs are stored locally by default
- Not validated for production or healthcare environments
